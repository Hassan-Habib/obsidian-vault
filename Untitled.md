#### **1. Description**

The MarkLogic administrative console fails to protect sensitive user-management actions against Cross-Site Request Forgery. Specifically, the endpoint responsible for creating new users and assigning security roles (`/add-user-go.xqy`) does not implement anti-CSRF tokens. Furthermore, it improperly allows state-changing operations via HTTP **GET** requests.

#### **2. Technical Impact**

An attacker can perform a **Full System Takeover** by:

1. Crafting a malicious URL that defines a new username, password, and the `admin` role.
    
2. Tricking an authenticated administrator into clicking a link or simply visiting a site where the URL is embedded as a hidden resource (like an `<img>` or `<iframe>`).
    
3. Once the request is processed, the attacker gains a persistent administrative back-door into the database.
    

---

#### **4. Root Cause Analysis**

- **Lack of Nonce/Token:** The request parameters are 100% predictable. There is no unique "secret" (nonce) required to validate that the request was intentional.
    
- **Violation of HTTP Standards:** The application uses `GET` for a "State-Changing" operation. `GET` requests are intended to be "safe" (read-only).
    
- **Broken SameSite Defenses:** Because this is a `GET` request, standard browser protections like `SameSite=Lax` often fail to block the cookie transfer if the victim navigates to the URL.
    

---

#### **5. Remediation**

1. **Mandatory POST:** Disable the ability to create or modify users via `GET`. Force all state-changing actions to use `POST`.
    
2. **Synchronizer Token Pattern:** Implement a unique, cryptographically secure anti-CSRF token (e.g., `_csrf_token`) that must be submitted and verified with every user-management request.
    
3. **Strict Cookie Attributes:** Set the `SessionID` cookie to `SameSite=Strict`.
    
4. **Origin/Referer Verification:** Ensure the server rejects requests that do not originate from the expected administrative domain.
